# Ch.1 키워드를 모르면 생기는 일 - WORD size 에피소드

[< 왜 CS를 알아야 하는가](./01-why-cs.md)

---

앞에서 사례를 세 개 들었다. 면접, 검색, AI. 전부 "키워드를 아느냐 모르느냐"의 차이라고 했다. 지금부터는 실제로 벌어진 이야기를 하나 하겠다. 이 강의의 작성자가 직접 겪은 일이다.


## 데이터가 깨진다

몇 년 전, 임베디드 시스템을 개발하던 시절이다. 펌웨어 팀에서 두 개의 프로세서 간에 데이터를 주고받는 기능을 만들고 있었다. 프로세서 A에서 구조체(struct)를 직렬화해서 보내면, 프로세서 B가 그걸 받아서 다시 구조체로 복원하는, 흔한 패턴이다.

<details>
<summary>직렬화 (Serialization)</summary>

메모리에 있는 데이터 구조를 바이트 열(byte stream)로 변환하는 과정이다.
네트워크로 데이터를 보내거나 파일에 저장하려면, 메모리 구조를 일렬로 펼쳐야 한다.
Python에서는 JSON 변환(`json.dumps`)이나 pickle이, 백엔드에서는 Protocol Buffers, MessagePack 등이 직렬화에 해당한다.
역직렬화(Deserialization)는 그 반대 과정이다.

</details>

그런데 프로세서 B에서 복원한 데이터가 깨졌다. 필드 값이 엉뚱한 숫자로 바뀌거나, 아예 0이 되는 경우도 있었다. 어떤 때는 정상이고, 어떤 때는 깨졌다. 재현 조건도 불규칙했다.


## 5명이 몇 주를 뒤졌다

팀원 5명이 디버깅에 매달렸다. 다들 경력 있는 엔지니어들이었다.

시도한 것들:

- 통신 프로토콜을 처음부터 다시 점검했다. 바이트 순서, 체크섬, 패킷 경계. 이상 없었다.
- 송신 측 직렬화 코드를 라인 단위로 검토했다. 로직에 문제가 없었다.
- 수신 측 역직렬화 코드도 한 줄씩 봤다. 마찬가지로 문제 없었다.
- 로직 분석기(Logic Analyzer, 하드웨어에서 전기 신호를 직접 들여다보는 장비)를 붙여서 물리적으로 오가는 비트를 직접 확인했다. 비트는 정확히 전달되고 있었다.
- 컴파일러 버그를 의심해서 컴파일러 버전도 바꿔봤다. 달라지지 않았다.

몇 주가 지났다. 모두가 지쳐가고 있었다.


## 원인: WORD size

어느 날, 한 엔지니어가 양쪽 프로세서의 데이터시트를 꼼꼼히 비교하다가 발견했다. 프로세서 A는 32비트 CPU였고, 프로세서 B는 16비트 CPU였다.

<details>
<summary>WORD size</summary>

CPU가 한 번에 처리하는 데이터의 기본 단위 크기다.
32비트 CPU의 WORD size는 32비트(4바이트), 64비트 CPU의 WORD size는 64비트(8바이트)다.
WORD size는 CPU의 레지스터 크기, 메모리 주소 체계, 데이터 정렬 방식에 영향을 준다.

여기서 중요한 것은 데이터 정렬(alignment)이다. CPU는 WORD size에 맞춰 정렬된 데이터를 가장 효율적으로 읽을 수 있기 때문에, C/C++ 컴파일러는 구조체의 필드를 WORD size의 배수 주소에 배치한다. 이때 필드 사이에 빈 공간(padding)이 삽입된다.

WORD size가 다르면, 같은 구조체 정의라도 필드 사이의 padding이 달라지고, 결과적으로 메모리 레이아웃이 달라진다.

</details>

C/C++ 컴파일러는 구조체의 각 필드를 CPU의 WORD size에 맞춰 정렬(alignment)한다. 필드와 필드 사이에 빈 공간(padding)을 넣어서, CPU가 효율적으로 읽을 수 있는 주소에 데이터를 배치하는 거다.

문제는, 32비트 CPU와 16비트 CPU에서 같은 구조체를 정의하면 padding이 다르게 들어간다는 거다.

Python만 써본 사람에게는 아래 코드가 낯설 수 있다. C 언어에서는 변수를 선언할 때 타입(`char`, `int`, `short`)과 크기를 명시한다. Python의 `x = 1`과 달리, C에서는 데이터가 몇 바이트를 차지하는지가 프로그래머의 책임이다.

예를 들어 이런 구조체가 있다고 하자:

```c
struct SensorData {
    char  type;     // 1바이트
    int   value;    // 4바이트 (이 예시 기준)
    short status;   // 2바이트
};
```

(참고: 실제로 `int`의 크기는 플랫폼마다 다를 수 있다. 16비트 CPU에서는 `int`가 2바이트인 경우도 많다. 여기서는 두 플랫폼 모두 `int`가 4바이트라고 가정하고, padding 차이에 집중한다.)

32비트 CPU에서의 메모리 배치:

```
offset 0: [type(1)] [pad(3)]                        -- int는 4의 배수 주소에 와야 하므로 3바이트 빈칸
offset 4: [value(4)]
offset 8: [status(2)] [pad(2)]                       -- 구조체 전체가 4의 배수여야 하므로 2바이트 빈칸
                                               총 12바이트
```

16비트 CPU에서의 메모리 배치:

```
offset 0: [type(1)] [pad(1)]                        -- int를 2의 배수 주소에 맞추면 1바이트 빈칸
offset 2: [value(4)]
offset 6: [status(2)]
                                               총 8바이트
```

<details>
<summary>구조체 정렬 (struct alignment/padding)</summary>

CPU가 메모리에서 데이터를 읽을 때, 특정 경계에 맞춰 정렬된 주소에서 읽는 것이 가장 효율적이다(혹은 그렇게만 읽을 수 있다). 그래서 컴파일러는 구조체 필드를 자연 정렬(natural alignment)에 맞춰 배치한다. 자연 정렬이란, N바이트 크기의 데이터를 N의 배수 주소에 배치하는 것이다. WORD size는 이 정렬의 상한선에 영향을 준다.

- `char` (1바이트): 아무 주소에나 배치 가능
- `short` (2바이트): 2의 배수 주소에 배치
- `int` (4바이트): 4의 배수 주소에 배치

이 규칙을 맞추기 위해 컴파일러가 필드 사이에 자동으로 삽입하는 빈 공간이 padding이다.

</details>

프로세서 A(32비트)에서 12바이트로 직렬화해서 보낸 데이터를, 프로세서 B(16비트)에서는 8바이트 구조체로 읽고 있었다. 바이트 오프셋이 어긋나니 필드 값이 엉뚱한 위치에서 읽혀서 깨진 거다.


## 교훈

이 버그를 잡는 데 5명이 몇 주가 걸렸다. 통신 프로토콜, 직렬화 코드, 하드웨어 신호까지 전부 뒤졌다.

그런데 만약 누군가 한 명이라도 "WORD size"라는 키워드를 알고 있었다면?

"잠깐, 두 프로세서의 WORD size가 다르면 struct padding이 달라지지 않나?"

이 질문 하나면 하루 만에 끝났을 버그다.

5명의 엔지니어가 몇 주 동안 뒤진 이유는 실력이 없어서가 아니다. 키워드를 몰랐기 때문이다. "WORD size"라는 키워드를 모르면, 구조체 정렬이라는 개념 자체가 떠오르지 않는다. 떠오르지 않으면 검색할 수도 없고, 의심할 수도 없다.

이게 이 강의가 "키워드"를 강조하는 이유다.

참고로 이 에피소드에서 WORD size의 기술적 세부사항까지 전부 이해할 필요는 없다. (크로스 플랫폼 환경에서는 WORD size 말고도 Endianness - 바이트 순서 - 같은 함정이 더 있다. 이 에피소드의 직접 원인은 아니었지만, "키워드를 몰라서 검토 목록에 올리지 못한 것"이라는 점에서는 같은 이야기다.)

중요한 건 "키워드를 아느냐 모르느냐"가 디버깅 시간을 몇 주에서 하루로 바꿀 수 있다는 사실이다.

"임베디드 이야기라 나한테는 해당 없는 것 아닌가?" 싶을 수 있다. 백엔드에서 WORD size 때문에 데이터가 깨지는 경우는 거의 없다. 하지만 같은 구조의 문제는 도처에 있다. Timezone을 모르면 시간 데이터가 3시간씩 밀리는 버그를 못 잡고, 문자 인코딩(UTF-8 vs EUC-KR)을 모르면 한글이 깨지는 원인을 못 찾는다. 코드 로직에는 문제가 없는데 데이터가 이상한 경우, 원인은 대부분 "모르는 키워드" 안에 숨어있다.
