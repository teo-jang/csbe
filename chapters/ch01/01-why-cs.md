# Ch.1 왜 CS를 알아야 하는가 - 세 가지 사례

[< 이 강의에 대해](./README.md)

---

CS(Computer Science)를 왜 알아야 하는가. 결론부터 말하면, 키워드를 모르면 문제를 풀 수 없기 때문이다. 검색도 못 하고, 동료에게 질문도 못 하고, AI에게 지시도 못 한다. 세 가지 사례로 보여주겠다.


## 사례 1: JD에는 CS가 없는데, 면접에서는 CS를 물어본다

아래는 가상의 백엔드 개발자 채용 공고(JD)다. 실제 공고들을 참고해서 만든 것이니 현실과 크게 다르지 않다.

```
[백엔드 개발자 채용]

자격요건:
- Python 또는 Java 1년 이상 경험
- FastAPI, Django, Spring Boot 중 하나 이상 사용 경험
- MySQL 또는 PostgreSQL 사용 경험
- AWS 환경 배포 경험
- Git 기반 협업 경험

우대사항:
- Docker, Kubernetes 사용 경험
- CI/CD 파이프라인 구축 경험
- 대규모 트래픽 처리 경험
```

CS 키워드가 하나도 없다. "프로세스", "스레드", "System Call", "인덱스" 같은 단어는 찾아볼 수 없다.

그런데 면접에 들어가면 이런 질문이 나온다:

- "프로세스와 스레드의 차이가 뭔가요?"
- "인덱스를 걸면 왜 빨라지나요?"
- "Connection Pool이 뭐고, 왜 필요한가요?"
- "async로 처리하면 항상 빨라지나요?"

<!-- 기대 키워드: 프로세스, 스레드, 인덱스, Connection Pool, async, I/O Bound, CPU Bound -->

왜 JD에는 없는 걸 면접에서 물어보는가?

JD는 "이 도구를 쓸 줄 아느냐"를 묻는 거고, 면접은 "이 도구로 문제가 생겼을 때 해결할 수 있느냐"를 묻는 거다. 도구는 가르칠 수 있지만, 문제 해결 능력은 CS를 모르면 안 된다.

운영 중인 서버가 느려졌다고 해보자. FastAPI를 쓸 줄 아는 건 도움이 안 된다. "이게 CPU Bound인가 I/O Bound인가", "Connection Pool이 고갈된 건 아닌가", "인덱스가 안 걸린 쿼리가 있는 건 아닌가" - 이런 키워드를 아는 사람만 원인을 추적할 수 있다.

<details>
<summary>JD (Job Description)</summary>

채용 공고에 명시된 직무 요구사항이다.
보통 자격요건(필수), 우대사항(선택), 업무 내용 등으로 구성된다.
JD에 적힌 기술 스택은 "이걸 쓸 줄 아느냐"를 묻는 것이지, "이걸 깊이 이해하고 있느냐"를 묻는 게 아니다.
면접에서 CS를 물어보는 이유가 바로 이 간극을 확인하기 위해서다.

</details>


## 사례 2: 같은 문제, 다른 검색어

금요일 오후 5시. 서버 모니터링 대시보드에 알람이 울린다. API 응답 시간이 평소 50ms에서 5초로 치솟았다.

주니어 A와 주니어 B가 동시에 문제를 본다.

주니어 A의 접근:

```
구글: "서버 느린 이유"
구글: "API 응답 느림 해결"
구글: "FastAPI slow response"
```

검색 결과는 온갖 잡다한 글이 쏟아진다. "서버 사양을 올려라", "캐시를 써라", "CDN을 적용해라"... 어디서부터 시작해야 할지 모른다.

주니어 B의 접근:

```
구글: "connection pool exhaustion FastAPI"
구글: "SQLAlchemy pool timeout"
```

2번째 검색에서 Stack Overflow 답변을 찾았다. DB Connection Pool의 기본 크기가 5인데, 동시 요청이 그보다 많아서 대기가 발생한 거다. `pool_size`를 늘리고 `max_overflow`를 설정하니 해결됐다.

<!-- 기대 키워드: Connection Pool, pool_size, timeout, 동시 요청 -->

차이가 뭔가? 둘 다 같은 문제를 봤다. 하지만 A는 "느리다"라는 현상만 알고, B는 "connection pool exhaustion"이라는 키워드를 알고 있었다.

키워드를 아는 것과 모르는 것의 차이는, "어디서부터 파야 하는지 아느냐"의 차이다.


## 사례 3: AI에게 같은 문제를 시키는 두 사람

2024년 이후 개발자의 상당수가 AI 코딩 도구를 쓴다. Claude, ChatGPT, Cursor, GitHub Copilot. 도구는 달라도 핵심은 같다 - AI에게 지시를 내려서 코드를 만들거나 문제를 해결한다.

그런데 같은 AI에게 같은 문제를 시켜도, 결과는 천차만별이다.

운영 중인 API의 응답이 느리다. 로그를 찍어보니 특정 엔드포인트에서 2초 이상 걸린다.

개발자 A의 프롬프트:

```
"이 API 성능 최적화 해줘"
```

AI의 응답: Redis 캐시 레이어를 추가하자, 응답을 gzip으로 압축하자, CDN을 적용하자... 방향이 너무 넓다. 뭘 먼저 해야 할지 모르겠다.

개발자 B의 프롬프트:

```
"이 API가 느린 게 I/O Bound 문제인 것 같다. DB 쿼리 로그를 봤는데
같은 테이블에 쿼리가 수백 번 나간다. N+1 쿼리인지 확인하고 해결 방법을 알려줘."
```

<details>
<summary>N+1 쿼리</summary>

목록 1개를 가져온 후, 목록의 각 항목에 대해 개별 쿼리를 N번 더 날리는 패턴이다.
예를 들어 게시글 100개를 조회한 뒤, 각 게시글의 작성자 정보를 가져오기 위해 쿼리를 100번 더 날리면 총 101번의 쿼리가 실행된다. 1번이면 끝날 걸 101번 하는 셈이다.
자세한 내용은 Ch.13에서 다룬다.

</details>

AI의 응답: 정확히 문제가 되는 ORM 코드를 찾아서 `joinedload`(한 번의 쿼리로 관련 데이터를 함께 가져오는 방식)로 변경하고, 쿼리 실행 계획까지 비교 분석해준다.

<!-- 기대 키워드: N+1, eager loading, explain analyze, ORM, query plan, I/O Bound -->

같은 AI다. 같은 문제다. 하지만 A는 AI가 어디로 가야 할지 모르는 지시를 내렸고, B는 문제의 영역(I/O Bound)을 좁히고 구체적인 현상(N+1)을 짚어서 AI가 어디를 파야 하는지 알려줬다. B도 처음부터 모든 걸 알고 있었던 건 아니다. "I/O Bound"와 "N+1"이라는 키워드 두 개가 방향을 잡아준 거다.

이 강의의 Part 2 (Ch.7~9)에서 이 주제를 본격적으로 다룬다. AI에게 올바른 지시를 내리려면, 결국 CS 키워드를 알아야 한다.


## 세 가지 사례의 공통점

| 상황 | 키워드를 모를 때 | 키워드를 알 때 |
|------|-----------------|---------------|
| 면접 | 꼬리 질문에 무너진다 | 원리까지 설명할 수 있다 |
| 검색 | "서버 느림"으로 헤맨다 | "connection pool exhaustion"으로 직행한다 |
| AI 활용 | "최적화 해줘"로 삽질한다 | "I/O Bound + N+1" 키워드로 방향을 잡는다 |

전부 같은 이야기다. 키워드를 아느냐 모르느냐.
